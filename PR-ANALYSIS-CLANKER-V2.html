Key Architectural Shifts

Database: Sync → Async

Every database caller in the entire codebase was updated. This is not a partial migration — all 30+ database functions are now async. This ripples through routes, middleware, ticket manager, and test suites.

Browser Interaction: Client → Server

Navigate & Scan moves browser rendering from the client (iframe) to the server (headless Chromium). The client now displays screenshots and sends coordinate-based interaction commands. This eliminates CORS, CSP, and SPA compatibility issues.

Integration Pattern: Single → Dual

The ticket manager now dispatches to both GitHub and Jira. The integration abstraction handles provider selection, and each provider implements its own deduplication, fingerprinting, and lifecycle management.

Section 07

Navigate & Scan Redesign

Why the Redesign

The Problem

The previous iframe-based approach was fundamentally broken for Single Page Applications. SPAs use client-side routing, which doesn't propagate across iframe boundaries. Additionally, iframes are blocked by CSP headers, X-Frame-Options, and CORS — making many modern sites untestable.

The New Approach: Screenshot-Driven Remote Browser

The solution moves all browser rendering to the server via Playwright. The client displays JPEG screenshots and sends interaction commands (click coordinates, text input, scroll direction). The server manages persistent browser sessions with full state (cookies, auth, local storage).

Session Lifecycle

  POST /api/ns/start { url, viewport, auth? }
    |
    +---> Launch headless Chromium
    +---> Create browser context (viewport, HTTP auth)
    +---> Navigate to URL, wait for networkidle
    +---> Take screenshot, return JPEG URL
    |
  [Interactive loop]
    |
    +---> POST /api/ns/click { sessionId, x, y }
    |       Coordinate scaling (display → viewport)
    |       Element identification via document.elementFromPoint()
    |       Parent walk (6 levels up) for interactive elements
    |       Selector generation (8-level priority)
    |       Returns: screenshot, currentUrl, elementInfo
    |
    +---> POST /api/ns/type { sessionId, selector, value }
    |       Multi-fallback: fill() → child input → keyboard.type()
    |
    +---> POST /api/ns/scroll { sessionId, direction }
    |
    +---> POST /api/ns/navigate { sessionId, url }
    |       Deep linking within same session
    |
    +---> POST /api/ns/scan { sessionId }
    |       Runs full visual QA pipeline on current page state
    |       Creates test record + auto-generates tickets
    |
  POST /api/ns/end { sessionId }
    +---> Close browser, delete screenshot directory

Intelligent Element Detection

When the user clicks a screenshot, coordinates are mapped to the actual page. The system then identifies and generates optimal selectors:

PriorityStrategyExample
1Unique ID#submit-btn
2data-testid[data-testid="login"]
3name attributeinput[name="email"]
4aria-label[aria-label="Close"]
5placeholderinput[placeholder="Search"]
6role + textbutton:has-text("Submit")
7tag + classbutton.btn-primary
8nth-child fallbackdiv > button:nth-child(2)

Additionally, the system walks up 6 parent levels to find interactive ancestors (button, a, input, select, label, role="button") and walks down into children to find inputs wrapped in container divs (common React pattern).

Session Management

  • Sessions stored in Map<string, NsSession> keyed by session ID
  • Each session tracks: browser, context, page, viewport, screenshot dir, creation time, last activity, recorded steps
  • Auto-cleanup every 60 seconds — sessions idle >30 minutes are force-closed
  • Screenshot directory cleaned up with session
  • Screenshots stored as /screenshots/ns/{sessionId}/{prefix}-{counter}.jpg

Section 11

Risk Assessment

HIGH

No Tests for Navigate & Scan API

The 598-line navigate-scan.ts has zero automated tests. It manages browser sessions, coordinates Playwright lifecycle, handles screenshot storage, and runs the visual QA pipeline. A regression here breaks the primary interactive testing feature. Recommend adding integration tests for session start/click/type/scan/end lifecycle.

HIGH

No Tests for Jira Integration

The 610-line jira.ts module handles external API calls, ticket deduplication, status transitions, and webhook processing — all untested. A bug in fingerprint generation could cause duplicate tickets at scale. A bug in the reopen logic could close valid tickets. Recommend unit tests with mocked Jira API responses.

HIGH

Database Migration: No Migration Script

The shift from SQLite to PostgreSQL has no migration tooling. Existing deployments on SQLite will need manual data migration. The schema is created via initDb() at startup, but there's no ALTER TABLE support for schema evolution. Consider adding a migration framework (e.g., node-pg-migrate) before this hits production with real data.

MEDIUM

Browser Session Memory Leaks

Each Navigate & Scan session spawns a headless Chromium instance (typically 200-400 MB RAM). Sessions auto-cleanup after 30 minutes, but a spike in concurrent sessions could exhaust server memory. The cleanup interval (60 seconds) means up to 60 seconds of zombied browsers after errors. Consider adding a max concurrent sessions limit.

MEDIUM

Jira Webhook Security

The /api/integrations/webhooks/jira endpoint accepts POST requests without signature verification. Jira Cloud supports webhook signatures (X-Hub-Signature), but this implementation doesn't validate them. An attacker could forge webhook payloads to trigger arbitrary re-scans. Recommend adding HMAC signature verification.

MEDIUM

Screenshot Storage Unbounded

Screenshots are stored on disk in /screenshots/ns/{sessionId}/. Directories are cleaned when sessions end, but if the process crashes mid-session, orphaned directories accumulate. There's no disk space monitoring or cleanup-on-startup. Consider adding a startup sweep for orphaned screenshot directories.

MEDIUM

Coordinate Mapping Precision

Click coordinates are scaled from display dimensions to viewport dimensions. If the frontend doesn't accurately report its display dimensions, clicks will land in wrong positions. Off-by-one or rounding errors in the coordinate math could cause missed clicks, especially on small UI elements like checkboxes or close buttons.

LOW

PostgreSQL Connection Pool Sizing

Max 10 connections with 30s idle timeout. Under moderate load this is fine, but if many concurrent scans run Jira + GitHub ticket creation (each requiring DB lookups and writes), the pool could become a bottleneck. Pool configuration isn't exposed via env vars. Consider making pool size configurable.

LOW

Jira Priority Field Compatibility

The Jira integration attempts to set priority on issues, falling back if the field is rejected (HTTP 400). This fallback is good, but the retry doubles the API call count for projects with non-standard priority schemes. Low risk since it's handled gracefully, but could slow ticket creation.

Section 12

Recommendations

Before Merge

  1. Add Navigate & Scan integration tests — at minimum: session start, click, type, scan, end. This is the highest-value test gap.
  2. Add Jira integration unit tests — mock the Jira API and test fingerprinting, deduplication, reopen logic, and webhook processing.
  3. Add Jira webhook signature verification — validate X-Hub-Signature header to prevent forged webhooks.

Before Production

  1. Add database migration tooling — node-pg-migrate or similar, to handle schema evolution without data loss.
  2. Add max concurrent sessions limit — prevent memory exhaustion from too many simultaneous Playwright browsers.
  3. Add startup cleanup — sweep orphaned screenshot directories and stale DB records on server boot.
  4. Make pool size configurable — expose PG_POOL_MAX as environment variable.

Nice to Have

  1. Frontend browser tests (Playwright or Cypress) for the app.js changes
  2. Rate limiting on Navigate & Scan endpoints (session creation is expensive)
  3. Disk space monitoring for screenshot storage
  4. Jira API rate limit handling (Jira Cloud has rate limits that aren't currently tracked)
