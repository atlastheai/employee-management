Section 01

Executive Summary — What Is This System?

"greg" is a real-time KPI intelligence dashboard branded as ATTIX. It tracks sales team performance across 3 brands (Vama, Vittori, TradeAlgo) by ingesting call data, SMS activity, tech demos, Calendly meetings, Zoho SalesIQ chat metrics, and GitHub developer activity — then uses Anthropic's Claude AI to generate daily executive-level performance analysis for each team.

What It Does in One Sentence

ATTIX ingests phone call, SMS, demo, chat, and code activity data from multiple sources via webhooks, aggregates it per sales agent, then runs it through Claude AI every morning at 7 AM PT to generate a performance brief that tells management who's hitting quota, who's slipping, and what to do about it.

Key Insight

Each brand has its own Pipedrive/CRM instance, its own set of agents with different quotas, and its own BDR dashboard page. But the AI intelligence engine, webhook infrastructure, and database structure are shared across all three. The code uses a "brand adapter" pattern — same logic, different model classes per brand.

Manual CSV Import (7 import routes)

The /import page allows manual CSV upload for bulk data loading. Available imports: Vama calls, Vittori calls, TradeAlgo calls, Vama SMS, Vittori SMS, TradeAlgo SMS, Vama demos, Vittori demos, and SalesIQ conversations. All imports use upsert logic (aggregate + deduplicate).

Google Apps Script Integration

Three Google Apps Scripts (vama_demo_sheet_script.js, vittori_demo_sheet_script.js, vittori_to_sheet_script.js) are installed in Google Sheets used by the sales teams. They push new rows to ATTIX webhooks on every sheet edit + reconcile all data every 5 minutes. This is how tech demo and TO data flows from Google Sheets into the dashboard automatically.

Bulk SQL Import

47 SQL batch files (vama_batch_001.sql through vama_batch_047.sql, totaling ~31 MB) contain historical Vama call data INSERT statements. These were used for initial data migration via batch_import.py and railway_import.py.

Section 06

AI Intelligence Engine (Anthropic Claude)

Core Technology

Uses Anthropic's Claude (claude-sonnet-4-6) via the api.anthropic.com/v1/messages API. Authentication via ANTHROPIC_API_KEY environment variable. Results are cached in the ai_analysis_cache database table.

How AI Analysis Works

  1. Data Collection: Query PostgreSQL for last 6 months of metrics per agent
  2. Pre-Computation: Calculate L7D/L30 totals, WoW/MoM trends, z-scores, quota attainment tiers, anomaly flags
  3. Prompt Construction: Build a detailed data summary with agent snapshots, then add role instructions (e.g., "You are a senior revenue analyst")
  4. Claude API Call: Send to Anthropic, receive structured analysis text
  5. Cache: Store result in ai_analysis_cache table for instant retrieval
  6. Serve: Dashboard pages display cached analysis, with background refresh triggered on page load

Output Format (AI Sections)

Each BDR analysis generates these standard sections:

  • THIS WEEK — Key numbers for the current period
  • MONTHLY PULSE — L30 trends vs prior period
  • TOP PERFORMERS — Who's exceeding quota and why
  • ACTION REQUIRED — Who's underperforming (by name, with numbers)
  • PATTERN INSIGHTS — Statistical anomalies, day-of-week patterns
  • RECOMMENDATIONS — Specific actionable steps for management

Interactive Q&A

The /ask/<brand> POST endpoint enables conversational follow-up questions on any brand's data. Users can ask things like "Why did John's connection rate drop?" and Claude responds with the cached analysis context + quota targets. Supports conversation history for multi-turn exchanges.

Scheduled Daily Refresh

APScheduler runs _scheduled_morning_refresh() at 7:00 AM Pacific Time every day. It sequentially regenerates all 6 brand analyses plus the homepage brief. This ensures the dashboard has fresh AI analysis ready when management checks it each morning.

Section 10

Strengths, Weaknesses, and Observations

Strengths

  • Real business value: This system is clearly in active use. It solves the real problem of "who's hitting quota and who isn't?" with AI-generated daily briefings.
  • Robust data ingestion: 12 webhook endpoints + CSV import + API polling + Google Apps Script integration. Covers most data sources automatically.
  • Deduplication everywhere: Almost every data entry path has dedup logic (unique constraints, upsert patterns, raw_datetime keys). Data integrity is well-handled.
  • Practical AI integration: Background thread pattern avoids Railway's 30s timeout. Caching ensures instant page loads. Morning cron ensures fresh data by start of business.
  • Interactive AI Q&A: Users can ask follow-up questions about any team's data — a genuine AI chatbot built on top of real performance data.
  • Hidden email management: Clean pattern for excluding admin/departed employees from dashboards and AI analysis.

Weaknesses / Technical Debt

  • Single-file monolith: 7,419 lines in one file (index.py). Models, routes, business logic, AI prompts, webhook handlers, admin functions — all co-located. Difficult to navigate, test, or refactor.
  • Hardcoded quotas: Agent quotas are Python dictionaries in source code. Adding/removing an agent requires a code deploy. Should be database-configurable.
  • Hardcoded credentials in defaults: Default values for DATABASE_URL, SECRET_KEY, Pipedrive API keys, and GitHub webhook secret are hardcoded as fallbacks in the source code. These should be required environment variables with no defaults.
  • No automated tests: Zero test files. A system tracking 30+ agents across 3 brands with complex quota logic has no test coverage.
  • 1 Gunicorn worker: Limits concurrent request handling. Background AI jobs block the thread pool. Under load, this could cause request queuing.
  • Fix/patch scripts accumulated: 20+ standalone Python scripts (fix_mobile_matrix.py, fix_layout_v2.py, apply_table_theme_v2.py, etc.) suggest iterative UI patching rather than systematic design.
  • No database migrations: Schema changes done via ALTER TABLE ... ADD COLUMN IF NOT EXISTS at startup. Fine for now, fragile at scale.

Relationship to ORGCOMMAND

Important Context for Carlos

"greg" (ATTIX) is already solving a subset of the ORGCOMMAND problem. It tracks per-agent KPIs (calls, connections, demos) with quota thresholds and AI-generated performance analysis. ORGCOMMAND extends this concept to cover all personnel (not just sales/BDR), adds cost tracking, contractor management, and termination workflows. The KPI engine architecture in ATTIX — webhooks → database → AI analysis → dashboard — is essentially the same pattern ORGCOMMAND will follow.

Summary

"greg" is a production-grade sales KPI intelligence dashboard that's actively used to monitor BDR performance across 3 brands. It ingests data from 6+ external systems via real-time webhooks, runs it through Claude AI every morning, and presents it in a polished matrix-themed dashboard. The architecture is pragmatic (single-file Flask app on Railway), the AI integration is well-engineered (background jobs, caching, interactive Q&A), and the data pipeline is thorough (12 webhooks + CSV + Google Apps Script). Technical debt exists (monolith, hardcoded quotas, no tests) but the system clearly delivers value in its current form.
