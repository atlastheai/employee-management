Section 01

Executive Summary

ORGCOMMAND is a CEO-owned command center that gives you complete visibility into who works across your 4 organizations, what they're doing, and whether they're productive. It replaces the current state — disconnected HR systems, untracked contractors, zero performance accountability — with a single dashboard where every person, every cost, and every KPI is visible at a glance.

The Problem

  • No unified view of headcount across companies
  • Multiple disconnected tools (Zoho, Pipedrive, Jira, GitHub, possibly Zenefits)
  • Contractors on ongoing retainers with zero delivery tracking
  • Cannot identify non-productive personnel for termination
  • Partial org chart — some teams have managers, others are flat
  • Losing money with no accountability mechanism

The Solution

A phased build of ORGCOMMAND that starts simple and grows:

  1. Phase 1 (Week 1-2): Import your roster via CSV. See everyone in one place.
  2. Phase 2 (Week 3-4): Add KPIs manually. Traffic-light dashboard. CEO uses it daily.
  3. Phase 3 (Week 5-7): Contractor tracking with retainer & deliverable monitoring.
  4. Phase 4 (Week 8-12): API integrations (Zoho, Pipedrive, Jira, GitHub). Data flows automatically.
Key Insight

You don't need to wait for integrations to get value. Phase 1 alone — just seeing all 200+ people with their costs in one table — is something you don't have today. We start manual, prove the value, then automate.

Core Design Principles

1. Email as Universal Identifier

Every person in ORGCOMMAND is identified by their email address. Since you confirmed emails are consistent across all systems, this becomes the join key between HR roster data, CRM metrics, Jira tickets, and GitHub commits. No fuzzy matching, no alias tables — clean 1:1 mapping.

2. Multi-CRM Unified Adapter

Zoho and Pipedrive have different APIs but serve the same purpose: tracking sales activity. We build a CRM Adapter Layer that normalizes both into a single data format:

  • deals_created — Number of new deals/opportunities
  • deals_closed — Number of won deals
  • revenue_generated — USD value of closed deals
  • pipeline_value — Total pipeline in USD
  • activities_logged — Calls, emails, meetings logged

The adapter maps Zoho's API fields and Pipedrive's API fields into these common metrics. The dashboard doesn't care which CRM the data came from — it just shows the numbers.

3. Manual-First, Automate Later

Phase 1 is entirely manual: CSV import for the roster, manual KPI entry for performance data. This is deliberate:

  • You get value in Week 1, not Week 8
  • You learn which KPIs actually matter before we automate the wrong things
  • Data quality is higher with manual entry than rushed integrations

4. Flexible Hierarchy

Since your org chart is partial (some teams have managers, others are flat), the data model treats the manager relationship as optional. A person can have a manager assigned or not. Teams can be managed or unmanaged. The system doesn't force a strict hierarchy.

5. USD Everywhere

All costs are stored in USD. When importing data from UK (GBP) or Singapore (SGD) teams, the importing user enters the USD equivalent. No exchange rate service, no currency conversion logic, no complexity. If you ever want to add original currency tracking, it's a simple schema extension.

Section 06

Contractor Management Solution

This is where ORGCOMMAND fills the biggest gap. Right now, contractors are on ongoing retainers, their deliverables aren't tracked, and you have no idea if they're delivering on time. Here's how we fix that.

The Problem

Contractors use company emails (good for matching), have ongoing retainers (not project-based), are "not really tracked," and you have "no idea if they delivered on time." This means you're potentially paying monthly retainers for work that isn't happening.

Retainer-Based Tracking Model

Unlike project-based SOWs, your contractors are on ongoing monthly retainers. The tracking model reflects this:

1. Retainer Record

For each contractor, capture:

  • Monthly retainer amount (USD)
  • Scope of work — Free-text description of what they're expected to do each month
  • Start date and end date (null if ongoing)
  • Review frequency — Monthly, quarterly, or custom

2. Monthly Deliverable Checklist

For each retainer, define what "delivered" looks like each month:

Deliverable Due Date Status Rating Notes
Monthly blog posts (4x) Last day of month Delivered 4/5 Good quality, on time
Social media calendar 5th of month Overdue - Not received
Weekly analytics report Every Monday Partial 2/5 Received 2 of 4 weeks

3. Contractor Performance Metrics

  • On-Time Delivery Rate — % of deliverables marked "Delivered" by due date
  • Average Quality Rating — Mean of 1-5 ratings across all deliverables
  • Cost per Deliverable — Monthly retainer / number of deliverables = effective cost per output
  • Consecutive Months Below Threshold — Ties into the same alert engine as employees

4. Contractor Traffic Light

Status Criteria Action
Green 80%+ deliverables on time, avg rating 3.5+ No action needed
Yellow 60-79% on time, OR avg rating 2.5-3.4 Flagged for review
Red <60% on time, OR avg rating <2.5 Escalation per threshold rules
Initial Setup

Since contractor deliverables aren't tracked anywhere today, Phase 3 starts with you (or a manager) defining what each contractor is supposed to deliver each month. This is a one-time setup per contractor. After that, monthly check-ins just mark things as delivered/not delivered.

Section 07

Performance Threshold System

You said: "I want to know immediately" but also "it depends." This means the system needs configurable thresholds — not a one-size-fits-all rule. Here's how we handle that.

Threshold Profiles

Instead of a single global rule, we create Threshold Profiles that can be assigned per role, per team, or per individual. This lets you be aggressive with some groups and forgiving with others.

Profile Name 1 Red Period 2 Consecutive Reds 3 Consecutive Reds Use For
Standard Flag in dashboard Email alert to CEO Termination recommendation Default for most roles
Aggressive Email alert to CEO Termination recommendation — High-cost, critical roles
Lenient Flag in dashboard Flag in dashboard Email alert to CEO New hires (ramp-up), seasonal roles
Custom (configurable) (configurable) (configurable) Per-individual exceptions

How It Works

Step 1: KPI Entry (Manual or Automated)

A KPI value is entered for a person for a given period. The system computes the traffic-light status based on the KPI template's thresholds:

  • Green: At or above target (e.g., 100%+ of goal)
  • Yellow: Below target but within tolerance (e.g., 70-99% of goal)
  • Red: Significantly below target (e.g., <70% of goal)

These percentages are configurable per KPI template.

Step 2: Alert Generation

When a red status is recorded:

  1. System checks the person's assigned threshold profile
  2. Checks how many consecutive red periods this person has
  3. Executes the appropriate action from the profile

Step 3: Escalation Path

PERIOD 1 RED: +-- Standard Profile ---> Dashboard flag (Action Required view) +-- Aggressive Profile -> CEO email alert +-- Lenient Profile ----> Dashboard flag PERIOD 2 RED (consecutive): +-- Standard Profile ---> CEO email alert + "Review Recommended" +-- Aggressive Profile -> Termination recommendation drafted +-- Lenient Profile ----> Dashboard flag (still monitoring) PERIOD 3 RED (consecutive): +-- Standard Profile ---> Termination recommendation drafted +-- Aggressive Profile -> (already recommended at period 2) +-- Lenient Profile ----> CEO email alert + "Review Recommended" AT ANY POINT: +-- CEO can override: dismiss alert, extend monitoring, or act +-- Green period resets the consecutive counter +-- Yellow period does NOT reset the counter (it's not green)

The "It Depends" Factor

Your instinct is right — context matters. The system supports this through:

  • Per-person profile overrides — Assign a new hire the "Lenient" profile for their first 3 months, then switch to "Standard"
  • Alert notes — When an alert fires, you can add context ("on medical leave", "client delayed project", etc.) and acknowledge it without dismissing
  • Manual override — CEO can always override any automated recommendation. The system advises; you decide.
  • Seasonal adjustments — Temporarily lower thresholds for known slow periods
Important

The system recommends termination — it never auto-terminates. Every recommendation goes through the Action Required view where you review context, history, and make the final call. The goal is to make sure nobody slips through the cracks, not to replace your judgment.

Email Matching Flow

When a connector pulls data from an external system:

  1. Extract the user's email from the external system record
  2. Look up persons.email in ORGCOMMAND
  3. If found: auto-create the KPI entry for that person/period
  4. If NOT found: add to a "manual review queue" for admin to resolve (typo? new hire? contractor using personal email?)
Your Advantage

Because everyone uses the same email across systems, matching is trivial. No alias tables, no fuzzy matching, no ML needed. This is the single biggest simplifier in the entire architecture.

Why Next.js Over Python/FastAPI?

The original MASTERPLAN proposed Python + FastAPI. We're recommending Next.js instead for these reasons:

  • Single codebase — No separate frontend and backend repos. API routes live alongside React components. Fewer deployment targets, fewer things to break.
  • Faster iteration — Full-stack changes in one PR. No "update the API, then update the frontend" two-step.
  • Better SDK ecosystem — Zoho, Pipedrive, Jira, and GitHub all have excellent Node.js libraries. Python equivalents exist but are less maintained.
  • Deployment simplicity — Railway deploys Next.js with zero configuration. One railway up and you're live.
  • Claude Code efficiency — Claude generates clean, idiomatic Next.js code rapidly. The framework's conventions reduce ambiguity.
Note

This is a recommendation, not a mandate. If you prefer the Python/FastAPI stack from the original MASTERPLAN, everything in this proposal works equally well with that stack. The architecture is tech-agnostic — only the implementation details change.
