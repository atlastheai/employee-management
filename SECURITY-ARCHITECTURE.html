Carlos's Concern — Directly Addressed

Your question: "How can we ensure that this data isn't accessible even to Atlas/OpenClaw, only to Claude Code during development?"

This is the right question. Here's the direct answer:

The Hard Truth

During development, Claude Code needs access to a database to build and test features. But that database will contain only fake/synthetic data. Your real employee data never touches the development environment. Here's how:

Three Separate Environments, Three Separate Databases

Environment Database Contents Who Has Access Purpose
Development 100% synthetic/fake data (auto-generated names, emails, salaries) Claude Code, Atlas Build and test features
Staging Synthetic data (larger volume, stress testing) Claude Code, Atlas Pre-production validation
Production Real employee data Carlos ONLY Live system with real data
Key Guarantee

Atlas and OpenClaw never see the production database credentials. Carlos creates the production database himself, sets his own password, and enters the credentials directly into the production environment variables. The developers never receive, see, or handle these credentials. The code deploys. The credentials are Carlos's alone.

How This Works in Practice

The Handoff Moment

When the application code is ready for production deployment, here's exactly what happens:

  1. Code is pushed to a GitHub repository (Carlos owns this repo)
  2. Carlos creates a Railway project (or provisions a VPS) using his own account
  3. Carlos creates the production PostgreSQL database using Railway's dashboard (or installs it on his VPS)
  4. Carlos enters environment variables himself — database URL, JWT secret, email API key — directly in Railway's dashboard. These values are never in the code, never in chat, never shared.
  5. Carlos deploys the code from GitHub to Railway with one click
  6. Carlos uploads his real CSV through the ORGCOMMAND web interface
  7. Done. The developers never had access to production credentials at any point.
About Claude Code Specifically

Claude Code is an AI assistant that runs locally in your terminal. It doesn't have persistent memory between sessions, doesn't phone home with your data, and doesn't store database credentials after a session ends. During development, it only interacts with the synthetic dev database. It has no mechanism to reach a production database it was never given credentials for.

Section 03

Database Security

Encryption at Rest

All data stored on disk is encrypted, meaning even if someone stole the physical hard drive, they couldn't read the data.

Deployment Option Encryption at Rest How It Works
Railway (recommended) Enabled by default Railway runs PostgreSQL on encrypted volumes (AES-256). Zero configuration needed. All data, indexes, and WAL files are encrypted on disk.
VPS (self-hosted) Manual setup Enable LUKS full-disk encryption on the VPS, or use PostgreSQL's pgcrypto extension for column-level encryption of sensitive fields.

Application-Level Encryption (Extra Layer)

Beyond database-level encryption, we encrypt the most sensitive columns at the application level before they're written to the database. This means even a database administrator (if one ever existed) couldn't read the values without the application's encryption key.

// Fields encrypted at the application layer before database storage // Uses AES-256-GCM with a key derived from an environment variable Encrypted fields: persons.monthly_cost_usd // Salary data persons.notes // Private performance notes performance_alerts.detail // Termination recommendations communications.body // Manager/CEO messages audit_log.detail // Detailed action data NOT encrypted (needed for queries/filtering): persons.first_name // Needed for search persons.last_name // Needed for search persons.email // Primary key for matching persons.company // Needed for filtering kpi_entries.status // Needed for traffic-light queries
How Application-Level Encryption Works

The encryption key (ENCRYPTION_KEY) is stored as an environment variable that only Carlos sets in production. The code uses this key to encrypt data before writing to PostgreSQL and decrypt when reading. Without this key, the encrypted columns are unreadable gibberish — even with full database access.

Encryption in Transit

Connection Encryption Implementation
Browser → App TLS 1.3 (HTTPS) Railway provides automatic SSL certificates. VPS: Let's Encrypt + nginx.
App → Database TLS/SSL PostgreSQL connection string includes ?sslmode=require. Railway enforces this by default.
App → External APIs TLS 1.2+ All API calls (Zoho, Pipedrive, GitHub, Jira) use HTTPS. No HTTP fallback.
// PostgreSQL connection string (production) // Carlos sets this in Railway's environment variables dashboard DATABASE_URL="postgresql://orgcommand:<CARLOS_SETS_THIS>@<PRIVATE_HOST>:5432/orgcommand_prod?sslmode=require" // The password is generated by Carlos or Railway // It is NEVER shared with developers // It is NEVER committed to source code

Database Access Controls

-- PostgreSQL role configuration (applied during production setup by Carlos) -- 1. Application role: can read/write app tables only CREATE ROLE orgcommand_app WITH LOGIN PASSWORD '<GENERATED>'; GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO orgcommand_app; REVOKE DROP, TRUNCATE ON ALL TABLES IN SCHEMA public FROM orgcommand_app; -- 2. Deployment Security

Railway Deployment (Recommended)

INTERNET | | HTTPS (TLS 1.3) v +-------------------+ | Railway Edge | <-- Automatic SSL certificate | (Public Endpoint) | <-- DDoS protection +-------------------+ | | Internal network (encrypted) v +-------------------+ | Next.js App | <-- Application container | (Private Service) | <-- No direct public access to app internals +-------------------+ | | Private networking (never exposed to internet) v +-------------------+ | PostgreSQL | <-- NO PUBLIC ENDPOINT | (Private DB) | <-- Only reachable from app container +-------------------+ <-- Encrypted at rest (AES-256) <-- Credentials known ONLY to Carlos

Railway Security Features Used

  • Private Networking — Database has no public hostname. It's only reachable from within Railway's private network. You cannot connect to it from the internet, a laptop, or any external tool.
  • No Public Database Endpoint — Railway allows disabling the public database URL entirely. We do this for production. The database is only reachable by the application.
  • Automatic SSL — HTTPS is automatic and enforced. HTTP requests redirect to HTTPS.
  • Environment Variables — Stored encrypted in Railway's vault. Not visible in logs. Not accessible via API without Railway account login (Carlos's account).
  • Deploy Previews — We disable deploy previews for production to prevent accidental exposure of the production environment.

VPS Deployment (Alternative)

If Carlos prefers a self-hosted VPS instead of Railway:

# Firewall rules (UFW on Ubuntu) # Allow HTTPS only ufw allow 443/tcp # Allow SSH only from Carlos's IP (for server management) ufw allow from <CARLOS_IP> to any port 22 # BLOCK all PostgreSQL external access ufw deny 5432/tcp # PostgreSQL listens on localhost ONLY # In postgresql.conf: listen_addresses = '127.0.0.1' # NOT '*' or '0.0.0.0' # SSH key authentication ONLY (no passwords) # In sshd_config: PasswordAuthentication no PermitRootLogin no

Environment Variable Security

All secrets are stored as environment variables, never in code:

Variable Contains Set By Visible To Developers?

JWT_SECRET 256-bit random secret for signing auth tokens Carlos NO
ENCRYPTION_KEY AES-256 key for application-level column encryption Carlos NO
SENDGRID_API_KEY Email service API key Carlos NO

Critical Rule

There is a .env.example file in the code with placeholder values (e.g., DATABASE_URL=postgresql://user:password@localhost/dev). This file shows what variables are needed but contains no real credentials. The real .env file is in .gitignore and is never committed. Carlos creates his own production .env or enters values directly in Railway's dashboard.

Database Backup Security

  • Automated backups — Railway: automatic daily snapshots, encrypted, retained for 7 days. VPS: cron + pg_dump + GPG encryption.
  • Backup encryption — All backups are encrypted before storage. Railway handles this automatically. VPS backups use Carlos's GPG key.
  • Backup access — Only accessible through Carlos's Railway dashboard or VPS SSH key. Developers have no access.
  • Backup restoration — Only Carlos can restore a backup. The app has no self-restore functionality exposed in the UI.
# VPS backup script (if self-hosted) — encrypted backup # Dump database pg_dump orgcommand_prod > /tmp/backup.sql # Encrypt with Carlos's GPG key gpg --encrypt --recipient carlos@company.com /tmp/backup.sql # Upload to private S3 bucket (Carlos's AWS account) aws s3 cp /tmp/backup.sql.gpg s3://orgcommand-backups/$(date +%Y%m%d).sql.gpg # Clean up unencrypted file shred -u /tmp/backup.sql

Authentication

Feature Implementation Details

Audit Logging

Every action in the system is logged to an immutable audit trail. This answers "who accessed what, when?" at any point in time.

// Sample audit log entries { "ts": "2026-03-15T09:23:14Z", "user": "carlos@acme.com", "role": "ceo", "action": "VIEW_PERSON", "target": "d.morrison@acme.com", "detail": "Viewed person detail page including salary" } { "ts": "2026-03-15T09:24:02Z", "user": "carlos@acme.com", "role": "ceo", "action": "EXPORT_CSV", "target": "all_personnel", "detail": "Exported full roster to CSV (214 records)" } { "ts": "2026-03-15T10:15:30Z", "user": "m.stevens@acme.com", "role": "manager", "action": "ENTER_KPI", "target": "j.smith@acme.com", "detail": "KPI entry: Deals Closed = 3 for Feb 2026" } { "ts": "2026-03-15T10:16:01Z", "user": "m.stevens@acme.com", "role": "manager", "action": "ACCESS_DENIED", "target": "salary_data", "detail": "Attempted to access salary data. Permission denied." }

Logged actions include:

  • Every login attempt (success and failure)
  • Every page/record viewed
  • Every data modification (create, update, delete)
  • Every CSV import
  • Every export/download
  • Every email sent from the system
  • Every permission denial
  • Every alert acknowledgment or dismissal

Session Security

  • Auto-logout after 30 minutes of inactivity — Configurable by Carlos
  • Single-session enforcement — CEO account can only be logged in from one device at a time. New login invalidates previous session.
  • Login notifications — Email sent to Carlos on every CEO login with IP address and device info
  • Failed login lockout — 5 failed attempts = 15-minute lockout + email alert to Carlos
  • IP logging — All logins record IP address for review

Compliance & Privacy

ORGCOMMAND stores personal data for employees across US, UK, and Singapore. While you're not legally required to comply with all regulations in a private internal tool, building to these standards is best practice and protects you if your data handling is ever questioned.

GDPR / UK GDPR Compliance Patterns

Applicable to your UK employees at Beta Inc:

GDPR Principle ORGCOMMAND Implementation
Lawful Basis Legitimate interest (employer managing workforce performance). Documented in a brief privacy notice for employees.
Data Minimization Only collect what's needed: name, email, role, cost, KPIs. No unnecessary personal data (no date of birth, no home address, no national ID).
Purpose Limitation Data is used exclusively for workforce management and performance evaluation. No secondary uses.
Storage Limitation Configurable retention policy. Terminated employee data can be auto-archived after X months.
Security Encryption at rest, in transit, application-level encryption, access controls, audit logging — all detailed in this document.

Right to Access / Right to Deletion

Data Subject Access Request (DSAR)

If an employee requests a copy of their data (their legal right under GDPR):

  • ORGCOMMAND includes an Export Person Data button on each person's detail page
  • Exports all stored data for that person as a JSON or CSV file
  • Includes: personal info, KPI history, performance notes, communications, alert history
  • CEO-only function — only Carlos can trigger this export

Right to Erasure ("Right to be Forgotten")

If an employee or ex-employee requests deletion:

  • Delete Person function permanently removes all personal data
  • Anonymizes audit log entries (replaces name/email with "DELETED_USER_[hash]")
  • Preserves aggregate statistics (team performance averages) without identifying individuals
  • CEO-only function with confirmation prompt
  • Logged in audit trail: "Person data deleted per DSAR/GDPR request"

Data Retention Policy

Data Type Default Retention After Retention Configurable?

Terminated employee records 12 months post-termination Auto-archived (anonymized) Yes
KPI history 24 months Aggregated (individual entries deleted) Yes
Audit logs 36 months Archived to encrypted cold storage Yes
Communication logs 12 months Deleted Yes

Singapore PDPA Compliance

For Gamma Systems employees in Singapore:

  • Similar principles to GDPR: consent, purpose limitation, access, correction
  • ORGCOMMAND covers these through the same mechanisms: export, delete, minimal data collection
  • No cross-border data transfer concerns if hosted in a US/EU data center (standard contractual clauses apply under PDPA Section 26)

Incident Response

If Carlos suspects unauthorized access:

  1. Immediately: Change database password in Railway dashboard
  2. Immediately: Rotate JWT_SECRET (invalidates all active sessions)
  3. Within 1 hour: Review audit log for unauthorized access
  4. Within 24 hours: Change all environment variable secrets
  5. If confirmed breach: Enable Railway's "private networking only" mode (disables public access entirely while investigating)
Nuclear Option

Railway allows you to instantly disable public access to the entire application with one toggle. If something feels wrong, flip that switch first, investigate second. Your data stays safe behind the private network while you figure out what happened.
